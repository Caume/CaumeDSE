# CaumeDSE Cryptography and Data Security Tutorial

This tutorial explains core cryptographic and data security concepts by using
CaumeDSE as an applied example. It is written for developers and operators who
want to understand why CaumeDSE stores and serves data the way it does, what
security properties those mechanisms provide, and what responsibilities remain
outside the engine.

The tutorial is not a formal security proof. It connects common security
concepts to concrete CaumeDSE features, source files, API routes, and verifier
coverage so that readers can audit and operate the system with better context.

## Learning Path

Read the sections in this order if you are new to data security:

1. Threat models and security boundaries.
2. Keys, salts, and key derivation.
3. Confidentiality with encryption.
4. Integrity with hashes, HMACs, and MAC-protected values.
5. Secure storage for CSV and raw documents.
6. Protected lookup indexes and searchable encrypted metadata.
7. Transport security, authentication, and authorization.
8. Parser execution, temporary files, deletion, and audit logs.
9. Verification and operational checklist.

## Threat Models and Boundaries

A threat model is a concrete statement of what you are defending, from whom,
and under which assumptions. CaumeDSE is designed to help store and process
sensitive data in environments where storage, transport, or hosting
infrastructure may be less trusted than the organization that owns the data.

CaumeDSE mainly helps with these risks:

- Offline disclosure of protected internal database values and secure document
  parts.
- Accidental broad access to resources through role tables and filter lists.
- Undetected tampering with protected data where HMACs are checked.
- Cleartext exposure in ordinary uploaded document storage.
- Loss of traceability for authenticated web requests.

CaumeDSE does not remove these responsibilities:

- Protecting organization keys outside the engine.
- Hardening the host, filesystem permissions, backups, swap, crash dumps, and
  administrator accounts.
- Validating OAuth or other external identity protocols when an engine manager
  is used.
- Auditing custom parser scripts before executing them.
- Choosing deployment-specific TLS certificates, reverse proxies, monitoring,
  and incident-response processes.

The most important boundary is the organization key. CaumeDSE uses `orgKey`
to protect resources for an organization, but it intentionally does not store
cleartext organization keys as a secret manager would. If an attacker obtains a
valid organization key and enough protected storage, they can attempt to
decrypt data for that organization.

## Keys, Salts, and Key Derivation

A symmetric encryption algorithm needs a binary key and often an initialization
vector. Human passwords are not normally suitable raw keys, so software derives
cryptographic key material from passwords through a password-based key
derivation function.

CaumeDSE routes encryption through OpenSSL EVP. The default symmetric cipher is
`AES-256-GCM`, configured by `cmeDefaultEncAlg` and described in `README.md`.
The key derivation path is implemented in `crypto.c` through `cmePBKDF()`.

Important concepts:

- `orgKey` is the organization-level secret provided with API requests.
- Each protected value or file part uses a salt so repeated plaintext does not
  normally produce the same protected value.
- `cmeDefaultPBKDFVersion` defaults to PBKDF2.
- `cmeDefaultPBKDFCount` controls the default PBKDF2 iteration count.
- If a supplied key is already a sufficiently long hexadecimal binary key,
  CaumeDSE treats it as high-entropy key material and uses the optimized
  derivation path documented in `README.md`.

Applied example:

```text
Client request:
  userId=EngineAdmin
  orgId=EngineOrg
  orgKey=<organization secret>

CaumeDSE:
  validates authorization metadata for the request
  derives encryption/MAC keys from orgKey and salts
  decrypts only the protected values needed for the operation
```

Operational guidance:

- Use high-entropy organization keys generated by a cryptographic random number
  generator.
- Treat `orgKey` as a secret with the same care as a database encryption key.
- Do not place organization keys in shell history, access logs, browser history,
  or monitoring systems.
- Prefer HTTPS and client certificate authentication for real deployments.

## Randomness and Salts

Randomness supports salts, generated IDs, temporary filenames, and other
security-sensitive values. CaumeDSE seeds OpenSSL through `RAND_poll()` and
checks `RAND_status()`. When `/dev/random` or `/dev/urandom` is available, the
engine can load additional entropy through OpenSSL.

Relevant source:

- `crypto.c`: `cmeSeedPrng()`, `cmePrngGetBytes()`,
  `cmeGetRndSalt()`, `cmeGetRndSaltAnySize()`.
- `common.h`: salt-size constants such as `evpSaltBufferSize`,
  `cmeDefaultSecureDBSaltLen`, and `cmeDefaultValueSaltLen`.

Security idea:

Salts do not need to be secret. They need to be unpredictable or unique enough
for their purpose. CaumeDSE stores salts alongside protected values so the same
organization key can derive the correct key material later.

## Confidentiality with Encryption

Encryption provides confidentiality: it makes data unreadable without the
right key. CaumeDSE uses OpenSSL EVP ciphers through wrappers such as
`cmeProtectByteString()` and `cmeUnprotectByteString()`.

The default cipher is `AES-256-GCM`. GCM is an authenticated encryption mode:
it encrypts data and produces an authentication tag that helps detect
tampering. CaumeDSE sets and verifies GCM tags in the cipher path when the
selected cipher is a GCM cipher.

Applied examples:

- Internal protected database values are encrypted before storage in
  ResourcesDB, RolesDB, LogsDB, and secure document metadata.
- Raw-compatible documents are split into parts, encrypted, and written as
  protected file parts.
- CSV documents are transformed into secure column-file databases instead of
  remaining as a plaintext CSV.

Relevant source:

- `crypto.c`: cipher initialization, PBKDF, GCM tag handling, byte-string
  protect/unprotect helpers.
- `db.c`: protected database value storage and retrieval.
- `filehandling.c`: raw file secure-file storage and secure CSV imports.
- `engine_interface.c`: protected register operations and secure DB deletion.

Security boundary:

Encryption protects stored data only while keys are unavailable to the
attacker. It does not protect data after the engine decrypts it to answer an
authorized request, run a parser script, or write a temporary response file.

## Integrity with Hashes, HMACs, and MAC-Protected Values

A hash maps data to a fixed-size digest. A plain hash can detect accidental
changes, but it does not authenticate who made the hash. An HMAC uses a secret
key and is appropriate for detecting tampering by someone who does not know
that key.

CaumeDSE defaults to SHA-256 for hashes and HMAC digest selection:

- `cmeDefaultHshAlg` is `sha256`.
- `cmeDefaultMACAlg` is `sha256`.

CaumeDSE uses HMACs in several places:

- Secure file part MACs.
- Protected database values stored as `MAC || ciphertext`.
- CSV column data attributes such as `MAC` and `MACProtected`.
- LogsDB transaction fields when authenticated request data is protected.

The distinction between MAC-before-protection and MAC-after-protection matters:

| Attribute | Meaning |
| --- | --- |
| `MAC` | MAC of the plaintext value before protection. Useful when verifying data semantics after decryption. |
| `MACProtected` | MAC of the protected value after encryption. Useful for detecting ciphertext-level tampering. |

Applied example:

When CaumeDSE retrieves protected database values, it can verify the HMAC
prefix before accepting decrypted data. That means an attacker who edits the
stored ciphertext without the organization key should be detected instead of
silently corrupting the returned value.

Relevant source:

- `crypto.c`: HMAC wrappers.
- `engine_interface.c`: protected DB register create/read/update/delete paths.
- `filehandling.c`: file part MAC generation and secure file reconstruction.
- `TEST/run_debug_components.sh`: `mac_macprotected` and HMAC/PBKDF markers.

## Secure CSV Storage

CSV storage is a central CaumeDSE example because it shows multiple concepts
working together.

When a CSV is uploaded as `file.csv`, CaumeDSE can:

1. Parse rows and columns.
2. Store data in secure column-file SQLite databases.
3. Protect values with encryption when the `protect` attribute is enabled.
4. Shuffle or protect row-order metadata when the `shuffle` attribute is
   enabled.
5. Store metadata describing column names, protection attributes, MACs, and
   file parts.
6. Register the resulting protected document parts in ResourcesDB.

The API route is:

```text
POST /organizations/{org}/storage/{storage}/documentTypes/file.csv/documents/{document}
```

With multipart parameters:

```text
file=@payroll.csv
userId=EngineAdmin
orgId=EngineOrg
orgKey=<organization secret>
shuffle=1
protect=1
replaceDB=1
```

After upload, these routes read protected content through the engine:

```text
GET /organizations/{org}/storage/{storage}/documentTypes/file.csv/documents/{document}/content
GET /organizations/{org}/storage/{storage}/documentTypes/file.csv/documents/{document}/contentRows/1
GET /organizations/{org}/storage/{storage}/documentTypes/file.csv/documents/{document}/contentColumns/name
```

Important security point:

The output of these routes is decrypted data. Transport security, response
handling, access control, and client storage matter as much as the encrypted
storage format.

## Raw Documents and Secure Files

Raw-compatible document types include formats such as `file.raw`, `file.txt`,
`file.json`, `file.xml`, `file.pdf`, images, archives, binaries, and script
documents. CaumeDSE stores these as secure files rather than secure CSV
databases.

The secure-file flow in `cmeRAWFileToSecureFile()`:

1. Reads the uploaded file.
2. Splits it into parts.
3. Generates a random part name and salt for each part.
4. Encrypts each part with the organization key and default cipher.
5. Computes an HMAC over the protected part.
6. Registers part metadata in ResourcesDB.

This gives confidentiality and tamper detection for stored file parts. It
also means that reconstruction requires both the protected part files and the
matching ResourcesDB metadata.

## Protected Lookup Indexes

Encrypted metadata is harder to search because equality comparisons normally
require decrypting every candidate record. CaumeDSE addresses this for selected
document fields with protected lookup indexes.

CaumeDSE computes deterministic keyed HMAC lookup values for:

- `documentId`
- `storageId`
- `orgResourceId`

The lookup input includes a domain string, the column name, the value length,
and the value. The result is keyed with the organization key. This allows
exact-match filtering to narrow candidates without storing plaintext lookup
columns.

Security tradeoff:

- Benefit: faster exact-match lookup on encrypted metadata.
- Residual leakage: deterministic indexes reveal when the same protected field
  value is queried or stored under the same key and domain.
- Mitigation: lookup values remain keyed and domain separated; CaumeDSE still
  decrypts and verifies candidate records before returning results.

Relevant source:

- `db.c`: `cmeGetProtectDBLookupValue()`.
- `engine_interface.c`: lookup-assisted protected DB query construction.
- `README.md`: document resource lookup behavior.

## Authentication, TLS, and Authorization

Authentication answers "who is making the request?" Authorization answers
"what may this authenticated identity do?"

CaumeDSE supports TLS client certificate authentication when HTTPS is used.
The client certificate chain is expected to include a user certificate and an
organization certificate. The authentication code verifies the chain and checks
that:

- the user certificate common name matches `userId`;
- the user certificate organization matches `orgId`;
- the issuer organization certificate common name matches `orgId`.

Relevant source:

- `webservice_interface.c`: `cmeWebServiceClientCertAuth()`.
- `TEST/run_debug_components.sh`: generated client certificate chain for live
  HTTPS verification.

Authorization uses protected resource tables:

- `roleTables` define allowed methods for resource types.
- `filterWhitelist` can restrict allowed resource patterns.
- `filterBlacklist` can deny matching resource patterns.

Applied example:

```text
POST /organizations/ExampleOrg/users/Alice/roleTables/documents
  *_get=1
  *_post=0
  *_put=0
  *_delete=0
  *_head=1
  *_options=1
```

This grants Alice read-like access to document resources but does not grant
document creation, modification, or deletion. Whitelist and blacklist entries
can further narrow or deny access to resource IDs matching regular
expressions.

Operational warning:

DEBUG builds may enable HTTP authentication bypass behavior for testing. Do
not treat DEBUG HTTP behavior as a production authentication model.

## Parser Scripts and Controlled Decryption

Parser scripts are an applied example of controlled decryption. The encrypted
CSV is reconstructed, then a script is run over the decrypted data. CaumeDSE
supports:

- `script.perl` through a child `perl` process that preserves the
  `cmePERLProcessColumnNames` and `cmePERLProcessRow` callback contract.
- `script.python` through `python3` with secure temporary input and output CSV
  files.

Parser route:

```text
GET /organizations/{org}/storage/{storage}/documentTypes/file.csv/documents/{document}/parserScripts/{script}
```

Security considerations:

- Parser scripts are executable code. Only trusted users should upload or run
  them.
- Parser output is decrypted application data.
- Temporary files must be protected by filesystem permissions and cleanup.
- Scripts should be reviewed for data exfiltration and resource exhaustion.
- Parser scripts run in child processes with absolute interpreter paths, a
  minimal environment, an explicit secure-temporary working directory,
  redirected stdio, closed inherited file descriptors, a compile-time timeout,
  an output-file size cap,
  and OS resource limits for CPU time, file size, address space, open files,
  and process count where supported. Perl and Python parser outputs share the
  parser result-table cap. The child processes still inherit the service
  user's filesystem and network privileges, so stricter filesystem and network
  sandboxing remain separate hardening work.

Prompt-injection boundary:

In an LLM-connected workflow, decrypted CSV cells and parser output are
untrusted text. A row can contain instructions such as "ignore your policy",
"send the organization key", or "load another file". CaumeDSE will store and
return that text as data; the application that sends it to an LLM must keep it
inside a data field and must not let it override system, developer, security,
or cleanup instructions.

For generated parser scripts, review the source before uploading it as
`script.perl` or `script.python`. Reject scripts that open network sockets,
execute shell commands, read environment variables, traverse files outside the
provided CSV input path, disable redaction, log credentials, or produce
unbounded output. Prefer deterministic transformations that use CSV parsing
libraries, select columns by exact header names, validate missing columns, and
write output through a CSV writer.

Verifier coverage:

`TEST/run_debug_components.sh` uploads Perl and Python parser scripts, runs both
over live HTTP and HTTPS flows, and verifies timeout and oversized-output
failure paths for both runtimes.

## Secure Deletion and Temporary Files

Deleting a file by unlinking it removes a directory entry, but storage media,
filesystems, snapshots, and SSD wear leveling may leave recoverable copies.
CaumeDSE provides a best-effort overwrite-and-delete helper for files it
controls.

Relevant source:

- `filehandling.c`: `cmeFileOverwriteAndDelete()`.
- `common.h`: `CDSE_SECURE_OVERWRITE_PASSES`.

By default CaumeDSE uses one overwrite pass. Builds can set more passes with:

```sh
CFLAGS="-DCDSE_SECURE_OVERWRITE_PASSES=3" ./configure
```

Operational boundary:

Secure overwrite is best effort. Use encrypted filesystems, controlled backup
retention, secure temporary directories, and storage lifecycle policies for
stronger guarantees.

## Audit Logs and Traceability

CaumeDSE logs web transactions in LogsDB. For authenticated requests, many log
fields are protected with the same protected database value pattern used
elsewhere: encryption plus HMAC prefixing. The `authenticated` flag remains
available for request classification.

Traceability helps answer:

- who requested a resource;
- which method and URL were used;
- when the request started and ended;
- which response code and size were produced.

Security boundary:

Logs can be sensitive. They may contain resource identifiers, request URLs,
and operational metadata. Protect log storage, backups, and administrative
access.

## Verification as a Security Practice

Security features need repeatable verification. CaumeDSE includes DEBUG
component and live API verification in `TEST/run_debug_components.sh`.

Useful commands:

```sh
# Full DEBUG verifier, including build and live HTTP/HTTPS API flows.
TEST/run_debug_components.sh

# Component-only rerun using an existing install.
TEST/run_debug_components.sh --skip-build --skip-web

# CI smoke profile: build, component markers, web startup, and HTTP live flow.
TEST/run_debug_components.sh --ci-smoke

# Focused live HTTP rerun.
TEST/run_debug_components.sh --live-only --web-protocol=http

# Focused live HTTPS rerun.
TEST/run_debug_components.sh --live-only --web-protocol=https
```

The verifier checks cryptographic primitives, protected database behavior,
role/filter resources, document routes, secure CSV round trips, MAC-protected
values, web startup, live API flows, parser execution, DB browsing, and
negative routes. The live API flow asserts missing-credential failures on HTTP
and HTTPS plus missing and mismatched client certificate failures on HTTPS.
Live API runs write `live-api-coverage.csv` and `live-api-coverage.txt` under
the verifier log directory so route coverage, expected/actual statuses, marker
checks, elapsed time, and response log paths can be reviewed without reading
the shell script.

For CI logs or AI-assisted debugging, run the verifier with
`CDSE_VERIFY_REDACT=1`.  This masks `orgKey`, `newOrgKey`, selected
credential-style request parameters, and generated certificate/key paths in the
run summary, live request artifacts, the full DEBUG run log, component extract
logs, and live service logs while preserving status codes, markers, elapsed
times, and artifact names.

## Operational Checklist

Use this checklist before handling real sensitive data:

- Generate high-entropy organization keys and store them outside CaumeDSE.
- Run production traffic over HTTPS with certificate management appropriate to
  the deployment.
- Disable testing-only authentication bypasses outside DEBUG verification.
- Place CaumeDSE data directories on protected storage with restricted
  permissions.
- Protect backups, snapshots, logs, and temporary directories.
- Use roleTables plus whitelist/blacklist filters for least privilege.
- Review parser scripts before upload and restrict script-authoring roles.
- Run `TEST/run_debug_components.sh` after security-sensitive changes.
- Document who can access organization keys, storage paths, and deployment
  certificates.

## Summary

CaumeDSE combines standard cryptographic building blocks with resource-based
authorization and repeatable verification. The strongest pattern in the system
is layered protection: encrypt stored values, authenticate protected data,
avoid plaintext lookup columns where practical, enforce authorization before
decryption, use TLS for transport, and keep auditable logs.

The system is still part of a larger operational security design. Strong
cryptography does not replace key management, host hardening, least privilege,
backup controls, script review, or incident response. CaumeDSE gives you
security mechanisms; secure operation depends on using them deliberately.
